# Tianxuan V5.0 — 极简内核发布

> 发布日期: 2026-06-05 | 基础版本: V4.1 | 净减代码: 3,900 行

---

## 1. 架构演进

```
V4.0c: 双模型 (Planner→Executor) + Learner反馈 + LLM Compact + 四域 TCCA
V4.2: 单模型 + 紧凑工具描述 + 流式预执行 + 文件缓存
V5.0: 极致单模型 + 零 LLM 辅助调用 + 纯截断替代 Compact
```

### V5.0 核心设计哲学

```
三原则:
1. 系统提示 ≤ 500 tok（含身份 + 规则 + 所有工具描述）
2. 零额外 LLM 调用（没有 Planner、没有 compact 摘要、没有 Learner）
3. 纯截断替代压缩（1M 窗口够用，截断免费）
```

## 4. 数据流 (单次 Turn)

```
用户输入
  │
  ├─ SQLite 查询相关记忆 (FTS5, < 5ms)
  ├─ 组装 Prompt: [System 500tok] [记忆 100tok] [历史] [输入]
  ├─ DeepSeek API (1 次调用, 无 Planner)
  │   ├─ 流式返回: reasoning → text → tool_calls
  │   └─ 流式预执行: 只读工具在 stream 期间并行完成
  ├─ 工具执行 (缓存命中直接返回, 写操作串行)
  └─ 结果注入对话历史
       └─ 若超 500K tok → 截断到最近 50 条消息
```

## 6. 关键设计决策

### 为什么删除 Planner?
- DeepSeek V4 推理能力已经覆盖了传统 Agent 的"规划层"
- 双模型路径增加 50% API 调用，但 GPT-3.5 时代的规划需求不再存在
- 1M 窗口意味着模型有足够的上下文自主规划

### 为什么删除 Learner?
- 自适应反馈循环从未被证实有效
- DeepSeek V4 行为一致，不需要学习纠偏
- 维护成本 > 运行时收益

### 为什么用截断替代 LLM Compact?
- DeepSeek 1M 窗口: 500 轮对话 ≈ 250K tok，远未触及上限
- LLM compact 本身消耗 token（摘要 ≈ ¥0.004），省下的 token 价值相同
- 截断是免费操作（< 1ms），compact 同步阻塞 5-30s

### 为什么保留 GitNexus?
- 代码图查询能力不可替代
- 本地运行，零 API 成本
- 提供了 grep/glob 无法实现的语义级代码理解
