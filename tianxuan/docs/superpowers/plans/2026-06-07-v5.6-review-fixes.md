## 任务 1：修复 maybeCompact — usage==nil 时静默跳过截断

**文件：** `internal/agent/compact.go`

**根因：** `maybeCompact` 第 35 行检查 `u == nil || u.PromptTokens == 0` 直接 return。当 provider 在某轮没有返回 usage（如 finish_reason="stop" 的空响应）时，永远不会触发截断。会话可能静默超出 1M 窗口。

**设计决策：** 当 `usage == nil` 时，使用 `LastPrompt`（上一轮已记录的 prompt 令牌数）作为 fallback。`LastPrompt` 在每轮正常结束时更新（第 51 行），所以它可以作为"已知最小 prompt 大小"的保守估计。如果上一轮已经接近阈值，本轮即使没有 usage 也应触发截断。

### 步骤 1.1：编写失败测试

在 `internal/agent/compact_test.go` 末尾追加：

```go
func TestMaybeCompact_NilUsage_FallbackToLastPrompt(t *testing.T) {
	a := &AgentRunner{
		compaction: CompactionConfig{Window: 100000, Ratio: 0.8, RecentKeep: 5},
	}
	// Build a session with many messages so truncation would fire if triggered.
	s := NewSession()
	s.Add(provider.Message{Role: provider.RoleSystem, Content: "system"})
	for i := 0; i < 100; i++ {
		s.Add(provider.Message{Role: provider.RoleUser, Content: "msg " + itoa(i)})
		s.Add(provider.Message{Role: provider.RoleAssistant, Content: "reply " + itoa(i)})
	}
	a.session = s
	a.compaction.LastPrompt = 90000 // 90% of window — above 80% trigger ratio

	// Call with nil usage — should still truncate because LastPrompt is high.
	a.maybeCompact(context.Background(), nil)

	// After truncation, messages should be reduced (system + RecentKeep tail).
	if len(a.session.Messages) >= len(s.Messages) {
		t.Fatalf("expected truncation when usage=nil but LastPrompt=90000 (above 80%% of 100k), got %d messages (was %d)",
			len(a.session.Messages), len(s.Messages))
	}
}

func (a *AgentRunner) maybeCompact(ctx context.Context, u *provider.Usage) {
	if a.compaction.Window <= 0 {
		return
	}
	// Use the provided usage when available; fall back to LastPrompt (the
	// last known prompt-token count) when the provider didn't return usage
	// this turn (e.g. finish_reason="stop" with no usage block). Session
	// growth is monotonic, so LastPrompt is a conservative floor.
	prompt := 0
	if u != nil {
		prompt = u.PromptTokens
	}
	if prompt == 0 {
		prompt = a.compaction.LastPrompt
	}
	if prompt == 0 {
		return
	}
	high := int(float64(a.compaction.Window) * a.compaction.Ratio)
	if prompt < high {
		a.compaction.LastPrompt = prompt
		return
	}
	// ... 后续截断逻辑不变 ...
```

## 任务 2：修复 toolcache TOCTOU 竞态

**文件：** `internal/agent/toolcache.go`

**根因：** `get()` 方法在 RLock 检查过期 → RUnlock → Lock 删除的间隙中，另一个 goroutine 可能通过 `set()` 添加了新的有效条目。随后的 `delete()` 覆盖了有效值。

### 步骤 2.1：编写失败测试

在 `internal/agent/toolcache_test.go`（新建或追加）中：

```go
func TestToolCache_TOCTOU_Race(t *testing.T) {
	// This test verifies that a set() between get()'s RUnlock and Lock
	// does not get erroneously deleted. We can't deterministically trigger
	// the race, but we can verify the fix by checking that after a
	// concurrent set+get cycle, all keys remain consistent.
	c := newToolCache(-1) // no TTL expiry
	tmp := t.TempDir()
	f := filepath.Join(tmp, "f.txt")
	os.WriteFile(f, []byte("hello"), 0644)

	c.set(f, 0, "hello")

	var wg sync.WaitGroup
	wg.Add(2)

	// Goroutine A: rapid get (which checks mtime, then re-locks)
	go func() {
		defer wg.Done()
		for i := 0; i < 100; i++ {
			c.get(f, 0)
		}
	}()

	// Goroutine B: rapid set with fresh content
	go func() {
		defer wg.Done()
		for i := 0; i < 100; i++ {
			c.set(f, 0, fmt.Sprintf("v%d", i))
		}
	}()

	wg.Wait()

	// After all operations, the cache should contain exactly one entry for this path.
	// With the TOCTOU bug, a get() checking TTL expiration could delete a fresh set().
	val, ok := c.get(f, 0)
	if !ok {
		t.Fatal("cache entry was incorrectly deleted by TOCTOU race")
	}
	if val == "" {
		t.Fatal("cache entry exists but has empty content")
	}
}
```

### 步骤 2.2：运行测试

```bash
go -C tianxuan test ./internal/agent/ -run TestToolCache_TOCTOU -v -race -count=5
```

### 步骤 2.3：实现修复

在 `get()` 方法中，将过期检查改为在写锁内双重检查：

```go
func (c *toolCache) get(path string, offset int) (string, bool) {
	key := readKey(path, offset)
	c.mu.RLock()
	ci, ok := c.items[key]
	c.mu.RUnlock()
	if !ok {
		return "", false
	}
	// Fast path: not expired. Skip the costly write-lock path.
	if c.ttl <= 0 || time.Since(ci.cached) <= c.ttl {
		// Still check mtime for freshness under read lock
		fi, err := os.Stat(path)
		if err != nil || !fi.ModTime().Equal(ci.mtime) {
			c.invalidatePath(path)
			return "", false
		}
		return ci.content, true
	}
	// Slow path: TTL expired. Re-check under write lock to avoid TOCTOU
	// with a concurrent set() that refreshed the entry between our RUnlock
	// and Lock.
	c.mu.Lock()
	ci2, ok2 := c.items[key]
	if ok2 && ci2 == ci {
		// Same entry — safe to delete.
		delete(c.items, key)
	}
	c.mu.Unlock()
	return "", false
}
```

## 任务 3：修复 session.Messages 直接引用

**文件：** `internal/agent/agent.go`，`stream()` 方法约第 610-622 行

**根因：** `msgs := a.session.Messages` 返回底层切片引用。当 L2 注入需要追加消息时，`injected := make([]provider.Message, 0, len(msgs)+1)` 创建新切片但 `append(injected, msgs[1:]...)` 可能在某些情况下与原切片共享底层数组。

**实际风险评估：** 审查 #5 说"append 在 cap 足够时直接修改原切片，可能导致并发读取 session.Messages 的 goroutine 读到损坏数据"。但仔细看代码：
```go
injected := make([]provider.Message, 0, len(msgs)+1)
injected = append(injected, msgs[0])
injected = append(injected, provider.Message{...})
injected = append(injected, msgs[1:]...)
```
`make([]provider.Message, 0, len(msgs)+1)` 创建了**全新**底层数组（cap = len(msgs)+1）。后续 append 都在新数组上操作——不会修改原切片。这个修复实际上是不需要的！

**重新评估：** 审查 #5 是误报。`make([]T, 0, cap)` 总是分配新数组，不会与任何现有切片共享。代码是正确的。

**行动：** 跳过此修复。在计划中记录为"已验证安全，无需修改"。

## 任务 4：修复 readStream goroutine 泄漏（#1）

**文件：** `internal/provider/openai/openai.go`

**根因：** `readStream` 的 `out` channel 是无缓冲的（`make(chan provider.Chunk)`）。在以下场景会发生泄漏：
1. agent.stream() 的 for range 因 ctx 取消而退出（ctx 取消 → resp.Body.Close() 被 goroutine 调用 → scanner 出错 → readStream 尝试发送 ChunkError）
2. 但此时 agent 端的 for range 已退出（ctx 被上游取消），无人接收 out channel
3. readStream 中的 `out <- ChunkError` 永久阻塞

**修复方案：** 给 out channel 添加缓冲（16 个 chunk），确保即使接收方提前退出，最后的错误发送也能写入缓冲区然后 goroutine 正常退出。同时，ctx.Done() goroutine 中添加安全关闭逻辑。

### 步骤 4.1：修改 `Stream()` 方法

在 `openai.go` 的 `Stream()` 方法中：
```go
// 原来：
out := make(chan provider.Chunk)

// 改为：
out := make(chan provider.Chunk, 16) // buffered so readStream can always exit
```

### 步骤 4.2：修改 `readStream` — 添加 ctx 检查

在 `readStream` 的 `scanner.Scan()` 循环中，每次写入 out 前检查 ctx：
```go
// 在 for scanner.Scan() 循环内部，每个 out <- 之前
select {
case <-ctx.Done():
    return // context canceled; defer close(out) will run
default:
}
```

实际上更简洁的方式是：在所有 `out <-` 发送前都检查 ctx。但这样改动太大。更简单的修复是：在发送 ChunkError 时使用 select：
```go
// 原来：
out <- provider.Chunk{Type: provider.ChunkError, Err: ...}

// 改为：
select {
case out <- provider.Chunk{Type: provider.ChunkError, Err: ...}:
case <-ctx.Done():
}
```

但这样错误会被吞掉。更好的方式是：带缓冲 channel 就足够了。receiving 端（agent.stream()）的 for range 只有在 channel 被 close 后才会退出。而 close 只在 readStream 的 defer 中执行。所以只要 readStream 能成功 close channel，agent 端就能正常退出。

问题是：如果 ctx 被取消，`resp.Body.Close()` 被 goroutine 调用，导致 scanner 出错。readStream 发送 ChunkError，然后 return。agent 收到 ChunkError 后 return。两者都正常退出。

**实际上，让我重新审视——审查指出的真正泄漏是什么？**

让我再看一遍代码流程：

1. `Stream()` 创建 `out := make(chan provider.Chunk)`（无缓冲）
2. 启动 goroutine: `go c.readStream(ctx, resp, out)`
3. `readStream` 中 `defer close(out)`
4. `readStream` 中 `go func() { <-ctx.Done(); resp.Body.Close() }()`
5. scanner 循环读取 SSE 事件...

如果一切正常：
- readStream 发送完所有 chunk + ChunkDone，然后 defer close(out) 执行
- agent.stream() 的 for range 正常退出
- 没有问题

如果 model 返回错误（如 JSON decode 失败）：
- readStream 发送 ChunkError，然后 return（defer close(out)）
- agent.stream() 收到 ChunkError，return（for range 退出，因为 channel 马上被 close）
- 没有问题

如果 ctx 被取消（在 stream 进行中）：
- ctx.Done() goroutine 调用 resp.Body.Close()
- scanner.Scan() 因 body 关闭而出错
- scanner.Err() 返回错误
- readStream 尝试发送 ChunkError 到 out
- agent.stream() 在等待 out channel
- 发送成功，然后两方都正常退出
- **没有问题！**

那泄漏在哪里？审查说"agent.stream() 可能在读完 channel 后提前 return，而 readStream 的后续 out <- 写入无人接收"。但 agent.stream() 的 for range 不会"提前 return"——它只在 channel 关闭时退出。而 channel 只在 readStream 的 defer close(out) 时关闭...

**等等，我发现了。** 如果 `agent.stream()` 中的 `for chunk := range ch` 循环内部因为某些原因 break 了（比如被外部的 context 取消触发了其他逻辑），那 channel 不会被完全消费。但看代码——for range 循环内部没有 break。唯一的退出路径是 ChunkError return 或正常结束。

**实际上我怀疑审查夸大了这个问题。** 让我重新评估：加缓冲 channel 是一个防御性改进，但不是严格必须的。真正的泄漏风险可能非常低。

但为了安全起见，我仍然加缓冲。这是一个低成本、高回报的防御措施。

### 步骤 4.1：修改

```go
// Stream() 中：
out := make(chan provider.Chunk, 16) // buffered: readStream can always push final error/close
```

## 任务 5：修复 ChunkError 后 pre-execution goroutines 泄漏（#2）

**文件：** `internal/agent/agent.go`，`runDirect()` 方法

**根因：** `runDirect()` 中，当 `stream()` 返回错误时（第 456 行）：
```go
text, reasoning, signature, calls, usage, err := a.stream(ctx, step+1)
if err != nil {
    return err  // ← 直接返回，preWG 的 goroutines 未 Wait
}
```
但 stream() 内部已经 `preWG.Add(1)` 启动了 pre-execution goroutines（第 693-698 行）。这些 goroutines 持有 `a.tc`、`a.preMu` 等共享状态——如果不等待它们完成就返回，后续 turn 可能碰到不一致的状态。

**修复：** 在 `stream()` 返回错误后，仍然调用 `a.preWG.Wait()` 清理。

### 步骤 5.1：编辑 runDirect()

在 `agent.go` 的 `runDirect()` 中：

```go
// 原代码（约第 454-460 行）：
text, reasoning, signature, calls, usage, err := a.stream(ctx, step+1)
if err != nil {
    return err
}

// 改为：
text, reasoning, signature, calls, usage, err := a.stream(ctx, step+1)
if err != nil {
    a.preWG.Wait() // drain any in-flight pre-execution goroutines
    return err
}
```

## 任务 6：删除 runParallel 外层空 recover

**文件：** `internal/agent/agent.go`，`runParallel()` 函数约第 807-815 行

**根因：** 外层 recover 是死代码。内层 `run(i)` 的 panic 已在第 790 行的闭包内被捕获并正确赋值了 `results[i]`。外层 recover 永远不会触发，其空函数体是误导性的。

### 步骤 6.1：删除死代码

```go
// 原代码：
go func() {
    defer wg.Done()
    defer func() { <-sem }()
    defer func() {
        if r := recover(); r != nil {
            // Surface the panic through the batch outcome channel
            // by storing it in the closures' results/outcomes vars.
        }
    }()
    run(i)
}()

// 改为：
go func() {
    defer wg.Done()
    defer func() { <-sem }()
    run(i)
}()
```
