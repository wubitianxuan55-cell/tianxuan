# V1.5 三层缓存重构 — 设计文档

> **状态**: 待审批
> **分支**: `v1.5-dev`（基于 `v1.4.0` tag）
> **回滚锚点**: `git checkout v1.4.0 -b rollback-v1.4`

---

## 1. 目标

| 指标 | V1.4 现状 | V1.5 目标 | Reasonix 参考 |
|------|----------|----------|--------------|
| 会话内缓存命中率 | ~94% | ≥99.3% | 99.8% |
| Miss rate | 6% | ≤0.7% | 0.2% |
| 跨会话冷启动 | Profile 每次重扫 | Profile 缓存复用 | N/A（无 Profile） |
| 子 Agent 缓存 | 继承被污染的 compiler | 继承纯净 L1 | 独立 prompt |

**30 倍 miss rate 差距的根因**：tianxuan V1.4 的 `compiler.go` 允许运行时修改 system prompt（`SetContextFocus`、`AddContextHint`），破坏 DeepSeek 前缀缓存的字节稳定性。

## 2. 架构：四域 → 三层

```
四域抽象                          三层缓存模型
─────────                        ─────────────
[Identity] 核心人设 ──────────┐
[Context]  项目画像固定部分 ───┤
           记忆索引 ──────────┼──→  L1 前缀缓存（完全不可变）
           技能索引 ──────────┘     每字节稳定 = DeepSeek 命中

[Context]  ContextFocus ──────┐
           FailureHint ───────┼──→  L2 运行时上下文（首轮后锁定）
[Skill]    GoalRouter 分类 ───┘     独立 system 消息，可被单独缓存

[Flow]     对话追加 ──────────┐
           Compaction L2 ring─┼──→  L3 对话流 + 持久化
           磁盘 index.json ───┘     追加式增长 + 低频压缩重置
```

### 2.1 L1 前缀缓存（完全不可变）

**职责**：每次 API 请求的 messages[0]，永远逐字节相同。

**内容**：
- Identity block（基础 system prompt）
- 语言策略
- 项目记忆（TIANXUAN.md + auto-memory index）
- 技能索引
- 工具描述（DeepSeek 独立字段，不影响 messages）

**规则**：
- ❌ 禁止任何 Set/Add/Append 方法修改 L1
- ❌ 禁止编译期锁（`sync.RWMutex`）——不需要，因为不可变
- ✅ 只在 `boot.Build()` 中构建一次
- ✅ 会话中任何运行时信息走 L2 或 L3

### 2.2 L2 运行时上下文（首轮后锁定）

**职责**：messages[1]，承载首轮分类后的运行时智能。

**内容**：
- ProjectProfile（文件数、模块名、目录结构）→ 首轮注入
- ContextFocus（Gather 模式匹配结果）→ 首轮分类后注入
- TaskKind 提示 → 首轮分类后注入
- FailureHint（FailureDetector 输出）→ 后续 turn 追加

**规则**：
- ⚡ 首轮：L2 为空消息（role: system, content: ""），无 token 开销
- 🔒 首轮 API 调用返回后，L2 内容锁定（GoalRouter 分类完成）
- 📌 后续 turn：L2 作为 messages[1] 发送，DeepSeek 可以缓存它（因为首轮后不变）
- 🔄 FailureHint 不是修改 L2，而是追加为 messages 中的独立 user 消息

**为什么 L2 也可以被缓存**：
```
Turn 1: [L1] [L2=""] [user: "fix bug"] ...  ← L2 空
Turn 2: [L1] [L2=focus+profile] [user: "fix bug"] [assistant] ... [user: "now refactor"]
         ↑L1 命中 ↑L2 未命中（内容变了）    ↑但 L2 从此稳定
Turn 3: [L1] [L2=focus+profile] [user: "fix bug"] [assistant] ... [user: "now refactor"] [assistant] ... [user: "add tests"]
         ↑L1 命中 ↑L2 命中！                 ↑history 前缀命中
```

### 2.3 L3 对话流 + 持久化

**职责**：动态增长的对话 + 压缩摘要 + 磁盘持久化。

**内容**：
- messages[2..N]：user → assistant(tool_calls) → tool(result) 循环
- Compaction L1 摘要（80% 窗口触发）
- L2 ring buffer（内存，最近 5 次压缩的详细上下文）
- 磁盘 index.json（跨会话回溯）

**规则**：
- 保持 V1.4 的 compaction 逻辑不变
- 保持 L2 ring + Learner + 记忆自动提取不变
- Compaction 是**故意的低频缓存重置点**（与 Reasonix 一致）

## 3. 代码改动清单

### 3.1 `internal/cache/compiler.go` — 简化为不可变编译器

```
改动：删除 mu, contextFocus, AddContextHint, SetContextFocus
保留：identity, context, registry, New, SystemPrompt, Assemble, Fork, FilteredSchemas
新增：无
```

`SystemPrompt()` 返回固定字符串，不再有任何可变状态。`Fork()` 简化为纯复制（无 contextFocus 继承）。

### 3.2 `internal/cache/runtime.go` — 新增 L2 运行时上下文

```
新文件：runtime.go
类型：RuntimeContext struct {
    profile      string  // ProjectProfile.Render() 输出
    taskKind     string  // GoalRouter 分类结果
    contextFocus string  // Gather 模式匹配
    locked       bool    // 首轮后锁定
}
方法：Inject(profile), ApplyRoute(cfg), IsLocked(), SystemPrompt()
```

`SystemPrompt()` 返回完整 L2 内容作为独立 system 消息。`locked=true` 后内容不再改变（保证 L2 缓存稳定性）。

### 3.3 `internal/cache/router.go` — 保持，输出对接 RuntimeContext

```
改动：Route() 仍返回 DomainConfig，但不再修改 compiler
新增：无
```

Controller 收到 DomainConfig 后调用 `runtime.ApplyRoute(cfg)` 而非 `compiler.SetContextFocus()`。

### 3.4 `internal/control/controller.go` — L1/L2 注入点

```
改动：
- runTurnWithRaw()：GoalRouter 结果 → runtime.ApplyRoute()（不再调 compiler.SetContextFocus）
- sendToAgent()：在 L1 和 user message 之间插入 L2 system 消息
- 删除 compiler.SetContextFocus 调用
```

### 3.5 `internal/agent/agent.go` — FailureHint 走 turn tail

```
改动：
- AddContextHint：不再调用 compiler.AddContextHint，改为往 session 追加 user 消息
- stream()：L2 消息插入在 system prompt 和对话历史之间
```

### 3.6 `internal/boot/boot.go` — 构建 RuntimeContext

```
改动：
- 创建 RuntimeContext，注入 ProjectProfile
- compiler 不再接收 profile 参数（profile 走 L2）
- 传给 controller 的 Options 增加 RuntimeContext 字段
```

### 3.7 `internal/cache/profile.go` — 保持，但输出目标变更

```
改动：无（Profile 扫描逻辑不变）
调用方变更：boot.go 中 profile.Render() → runtime.Inject(profile)
```
