## 十、新参考：MiMo-Code 关键设计

> 来源：https://github.com/XiaomiMiMo/MiMo-Code
> 定位：OpenCode 的增强分支，增加了持久记忆、智能上下文管理、子代理编排、目标驱动自主循环、自我进化能力

### 10.1 多 Agent 体系（build/plan/compose）

MiMo-Code 的三个 Agent 有不同的工具权限：
- **build**：完整工具权限，默认开发模式
- **plan**：只读分析模式，用于代码探索和方案设计
- **compose**：规范驱动开发，内置 planning/execution/review/TDD 等技能编排

**对 tianxuan 的启发：** 当前 tianxuan 只有 planMode on/off，不够细腻。V6.0 应该支持三种模式：
- explore（只读探索）= 当前 planMode
- develop（完整开发）= 当前默认模式
- orchestrate（编排模式）= 计划→执行→验证→合并 全流程

### 10.2 带预算的上下文注入（Budgeted Injection）

这是 MiMo-Code 最精华的设计。它不用 compaction 砍历史，而是用 token 预算控制注入量：

```
Context limit 100K:
- checkpoint:     30K（重要性 0.9）
- project memory: 20K（重要性 0.7）
- task progress:  10K（重要性 0.5）
- recent messages: 40K（重要性 1.0）
按重要性排序，超出预算的截断
```

**对 tianxuan 的启发：** 替代当前简单粗暴的 maybeCompact。当上下文接近窗口限制时：
1. 保存 checkpoint（tianxuan V5.31 已有此功能）
2. 按预算重新组装上下文（checkpoint + memory + task progress + 最近消息）
3. 按重要性排序注入

这样不会破坏缓存（前缀保留），同时上下文得到有效管理。这是 tianxuan V5.x 的 compaction 之后 V6.0 应该迁移的目标。

### 10.3 Dream & Distill（自我进化）

- **/dream**：扫描最近会话，提取知识到持久记忆，淘汰过时条目
- **/distill**：发现重复的人工工作流，打包为可复用的 skill/subagent/command

对 tianxuan：可以增加 /dream 和 /distill 命令，利用 LLM 自动发现模式和知识。

### 10.4 Goal / Stop 条件验证

独立 judge 模型评估"目标是否真的完成"，防止乐观提前停止。

对 tianxuan：/goal 命令 + judge 模型验证，防止任务做一半就假装完成。

## 十一、V6.0 最终路线图
