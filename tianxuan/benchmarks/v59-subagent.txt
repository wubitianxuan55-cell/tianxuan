## `cmd/tianxuan/main.go` 功能分析

### 1. 整体定位

`cmd/tianxuan/main.go` 是 **tianxuan 的主程序入口**——一个配置驱动、插件化的编码智能体 CLI。它编译后产生 `tianxuan` 二进制文件。

### 2. 文件结构（仅 20 行）

```go
package main

import (
    "os"
    "tianxuan/internal/cli"
    // 三个 blank import，编译期注入能力
    _ "tianxuan/internal/provider/anthropic"  // Anthropic（Claude）模型后端
    _ "tianxuan/internal/provider/openai"     // OpenAI 兼容模型后端（DeepSeek 等）
    _ "tianxuan/internal/tool/builtin"        // 内置工具集
)

var version = "dev"   // 通过 -ldflags "-X main.version=..." 在构建时注入

func main() {
    os.Exit(cli.Run(os.Args[1:], version))
}
```

### 3. 核心工作流程

```
main() → cli.Run(args, version)
           ├─ i18n.DetectLanguage()     // 检测/加载 UI 语言（中文/英文等）
           ├─ [无参数] → welcome()      // 首次运行引导 / 状态仪表盘
           ├─ "run"    → runAgent()     // 单次任务执行（非交互式）
           ├─ "chat"   → chatREPL()     // 交互式对话 REPL（BubbleTea TUI）
           ├─ "serve"  → runServe()     // HTTP/SSE 服务模式
           ├─ "setup"  → setupConfig()  // 配置向导
           ├─ "init"   → initHint()     // 项目记忆初始化提示
           ├─ "acp"    → acpCommand()   // Agent Communication Protocol
           ├─ "mcp"    → mcpCommand()   // MCP 服务器管理
           ├─ "codegraph" → codegraphCommand() // 代码图查询
           ├─ "doctor" → doctorCommand()// 系统诊断
           ├─ "version/-v" → 打印版本
           └─ "help/-h"   → 打印帮助
```

### 4. 三个 blank import 的作用

这是一套**注册表 + 编译期自注册**的插件架构：

| 包 | 注册的"kind" | init() 行为 |
|---|---|---|
| `provider/anthropic` | `"anthropic"` | 调用 `provider.Register("anthropic", factory)` — 注册 Claude 模型后端 |
| `provider/openai` | `"openai"` | 调用 `provider.Register("openai", factory)` — 注册 OpenAI/DeepSeek 兼容后端 |
| `tool/builtin` | 17 个内置工具 | 逐一调用 `tool.RegisterBuiltin(...)` — 注册 `read_file`、`write_file`、`bash`、`grep`、`edit_file` 等 17 个工具 |

这些包的 `init()` 在程序启动时自动执行，把能力注入全局注册表。后续 `cli.Run` → `boot.Build` 读取 `tianxuan.toml` 配置文件，按 `kind` 从注册表取出对应工厂函数，实例化模型后端。

### 5. 启动时装配链（boot.Build）

```
config.Load()          → 读取 tianxuan.toml
config.ResolveModel()  → 解析默认/指定模型
NewProvider()          → kind → registry[kind](cfg) → Provider 实例
tool registry          → 内置工具 + MCP 工具 + 插件工具
permission gate        → 权限策略（ask/allow/deny）
agent executor         → 单模型 / 双模型协调器（planner+executor）
→ control.Controller   → 暴露给前端（TUI / HTTP / 桌面）
```

### 6. 与项目中其他 `main.go` 的关系

| 入口 | 用途 |
|---|---|
| `cmd/tianxuan/main.go` | **主 CLI** — 完整智能体，含交互/非交互/服务三种模式 |
| `cmd/e2ebench/main.go` | **端到端基准测试** — 批量运行编码任务、评分、生成报告 |
| `cmd/tianxuan-plugin-example/main.go` | 插件开发示例 |
| `desktop/cmd/sign/main.go` | 桌面版代码签名工具（`sign`/`manifest`/`genkey`/`verify`） |

### 7. 关键架构特点

- **配置驱动**：模型、工具、权限全部由 `tianxuan.toml` 配置，代码里不硬编码具体模型名。
- **多前端共享同一装配**：`boot.Build` 是唯一组装点，TUI 聊天 (`chatREPL`)、HTTP 服务 (`runServe`)、桌面版 webview 都复用它。
- **国际化**：通过 `i18n` 包支持中英文 UI，语言可通过配置或 `setup` 向导选择。
- **插件化**：通过 Go 的 `init()` + 注册表模式，添加新 provider 或工具只需写包 + blank import。

## `internal/cache` 包结构分析

这是一个**四域前缀缓存（L1–L4）**系统，为 DeepSeek 服务端上下文缓存设计。它的核心思路是将提示词拆分为多个确定性字节域，每个域在会话间保持稳定，使服务端可以自动命中前缀缓存，避免重复处理。

---

### 📁 文件布局（7 源文件 + 4 测试文件）

| 文件 | 行数 | 职责 |
|---|---|---|
| `compiler.go` | 51 | L1 门面入口（已标记 Deprecated） |
| `identity.go` | 130 | L1 身份域：不可变系统提示 + ContentHash 跨会话验证 |
| `profile.go` | 190 | 项目扫描器：生成 Markdown 格式的项目快照 |
| `router.go` | 235 | 目标路由器：意图分类 → DomainConfig |
| `runtime.go` | 415 | L2 运行态域：项目/会话/执行状态追踪 |
| `skill.go` | 393 | L3 技能策略域：意图分类 + 版本化工具策略 |
| `spawn.go` | 272 | L4 子代理模板域：Fork 子代理 + 领域缓存 |

---

### 🏗️ 四域架构

```
L1 身份域 (Identity)     → 系统提示 + 工具注册表，整个会话不变
L2 项目域 (Runtime)      → 项目结构快照 + 编辑跟踪，支持 Markdown/KV 双格式
L3 技能域 (Skill)        → 意图分类(10种 TaskKind) + 版本化工具集(v1/v2/v3)
L4 子代理域 (Spawn)      → Fork 模板注册 + LRU 领域缓存(64条目)
```

### 🔗 核心关系

- **`Compiler`** 包装 `IdentityLayer`，是对外统一入口
- **`GoalRouter`** 绑定 `SkillLayer` + `Profile`，输入文本 → 分类 TaskKind → 过滤工具集
- **`RuntimeLayer`** 维护 `ProjectState`（Lock 后不可变）、`SessionState`（编辑追踪、执行上下文）
- **`SpawnPolicy`** 管理子代理 Fork，domainCache 做 SHA-256 哈希 → 同种子代理共享前缀命中 DeepSeek 缓存

### 💡 关键设计点

1. **前缀稳定性**：每个域输出确定性的字节序列，DeepSeek 服务端据此命中缓存
2. **跨会话验证**：`ContentHash()` 写入 `identity.hash`，下次会话 `LoadAndCompareHash()` 比对决定缓存是否有效
3. **LRU 淘汰**：`SpawnPolicy.domainCache` 上限 64 条，超限淘汰最旧 `FirstUsed` 条目，无 TTL
4. **V5.0 简化**：Learner 自适应升级已移除，`PromoteVersion()`/`RecordOutcome()` 均为空操作，改为纯静态版本管理
5. **无传统过期**：缓存生存期等于会话生命周期，由调用方 `context` 包负责清理

## `internal/boot/boot.go` 启动流程分析

核心入口是 **`Build()`**（第 67 行），整个启动是一个**单函数线性执行**的流程，最终返回 `*control.Controller`。没有显式的分阶段机制，但有两个后台异步操作（CodeGraph 安装、Plugin 子进程）不阻塞返回。

---

### 启动阶段总览

| 阶段 | 内容 | 错误策略 |
|---|---|---|
| **0** 配置加载 | 加载 `tianxuan.toml`、解析模型名、校验 API Key | 致命 |
| **1** 基础设施 | Sink（事件同步器）、JobManager、Provider 构造 | Provider 致命 |
| **2** System Prompt | 拼接基底 prompt → outputstyle → 语言策略 → 记忆 → skills 索引 | 致命 |
| **3** 缓存层 | 项目扫描、L1 编译缓存（前缀缓存）、L2 运行时层 | — |
| **4** 工具注册 | 内置工具 + CodeGraph + MCP Plugin + LSP | 全部非致命 |
| **5** 权限/钩子 | 权限策略、headless gate、项目 hooks、子代理工具 | hooks 非致命 |
| **6** 技能工具 | skill runner 闭包、子代理模板缓存、builtin 子代理工具 | — |
| **7** Agent 执行器 | ToolDispatcher、Session、主 Agent executor | — |
| **8** Slash Commands + AutoPlan | 命令加载、`/name` 工具、Plan 分类器 Provider | 分类器致命 |
| **9** Controller 组装 | GoalRouter、四层上下文管理器、缓存预热、`control.New()` | — |

---

### 关键设计决策

1. **L1/L2 双层缓存** — L1（IdentityLayer）是固定前缀可跨会话缓存，L2（RuntimeLayer）携带项目状态在首轮消息时注入。
2. **工具失败不阻塞** — CodeGraph、MCP Plugin、沙箱检测失败都只发 Warning Notice，降级运行。
3. **CodeGraph 三种路径** — 已安装直接注入 / 自动安装则后台下载 / 未安装仅提示。
4. **AutoPlan 分类器** — 如果启用且配置了独立模型，构造第二个 Provider，失败则致命终止。

---

- **CodeGraph Install**（L186-192）：`go codegraph.Install()` 后台下载，不阻塞 `Build()` 返回
- **MCP Plugin 子进程**：`plugin.StartAvailable()` 启动各 MCP server，失败通过 Notice 通知

### 🟡 中危 (Medium) — 3 项

#### 1. JSON 炸弹攻击面（DoS）

**位置：** 第 890、1157、1238、1268、1274、1334 行

`json.Unmarshal` 直接解析 LLM 返回的 `call.Arguments`，没有任何深度限制。LLM 是**不可信的数据源**——恶意 prompt 注入可使模型生成深度嵌套的 JSON（如 10 万层 `{"a":{"a":...}}`），导致：
- 内存耗尽（OOM）
- CPU 100% 占用
- 整个 agent 进程崩溃

```go
// 第 1157 行 — 无深度限制
if err := json.Unmarshal([]byte(call.Arguments), &raw); err == nil {
```

**建议：** 使用 `json.NewDecoder` 或预检查参数长度；对 `map[string]any` 的嵌套深度设置上限（如 20 层）。

---

#### 2. Panic 恢复中的敏感信息泄露

**位置：** 第 949-952 行

```go
defer func() {
    if r := recover(); r != nil {
        results[i] = fmt.Sprintf("tool panic: %v", r)
        outcomes[i] = toolOutcome{output: results[i], errMsg: fmt.Sprintf("panic: %v", r)}
    }
}()
```

`recover()` 捕获的 panic 值通过 `%v` 直接暴露给模型（进而暴露给用户）。工具执行时若发生 panic，可能包含：
- 文件系统绝对路径
- 数据库连接字符串
- 内部状态信息

**建议：** 对 panic 值做脱敏处理，至少过滤掉路径、URL、类凭证字符串后再输出。

#### 3. 缓存键路径未规范化（路径遍历）

**位置：** 第 1233-1243 行（读缓存）、第 1272-1276 行（写失效）

```go
// 第 1238 行 — ra.Path 直接来自 LLM，无 filepath.Clean
if err := json.Unmarshal(json.RawMessage(call.Arguments), &ra); err == nil && ra.Path != "" {
    if cached, ok := a.tc.get(ra.Path, ra.Offset); ok { ... }
}
```

路径来自 LLM 的工具调用参数，未经 `filepath.Clean` 处理直接用作缓存键。攻击者可通过 `foo/bar/../../etc/passwd` 与 `/etc/passwd` 造成缓存键不一致——同一文件被缓存两次，或缓存投毒。

**建议：** 在 `get`/`set`/`invalidatePath` 前对路径调用 `filepath.Clean`。

### 🟢 低危 (Low) — 3 项

#### 4. 生产环境使用 `panic()` 控制流程

**位置：** 第 1607 行


前缀指纹漂移时直接 `panic` 而非返回错误。文档注明这是刻意设计——"漂移 → panic，避免静默破坏缓存"。但在服务端部署中，一个 goroutine 的 panic 可能级联导致整个进程退出。

**建议：** 若此代码路径在服务器环境中运行，改为返回 error 并通过事件系统通知。
